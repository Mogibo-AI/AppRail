# AppRail ビジョンドキュメント

## エグゼクティブサマリー

AppRailは、AI-DLCの流れに沿って業務アプリ開発を支援するAI駆動型プラットフォームです。ユーザーが作りたいアプリのアイデアを入力すると、AppRailが要件整理、MVPスコープ定義、画面設計、データ設計、実装タスク分解、コード生成、テスト、AWSへのデプロイまでを支援します。

主な対象は、GitHub、AWS、CI/CD、環境変数、データベース設計などに詳しくない非エンジニアです。MVPではまず社内向けCRUDアプリの作成に絞り、要件整理からデプロイまでの一連の開発フローを確立します。その後、業務ポータル、予約・申請ワークフロー、分析ダッシュボード、顧客向けWebアプリなど、より幅広いアプリ種別へ展開していきます。

## 解決したい課題

非エンジニアは、解決したい業務課題や作りたいアプリのイメージを持っていても、それを動くアプリにするための要件定義、設計、実装、テスト、デプロイを自力で進めることが難しいです。

一方で、AIに単純にコード生成を依頼すると、前提の誤解、スコープの拡大、危険な変更、テスト不足、無承認のデプロイなどが起こる可能性があります。AppRailは、AIに作業を任せながらも、人が重要な判断点を確認・承認できる安全な開発レールを提供します。

## 対象ユーザー

| ユーザー種別 | 説明 | 主なニーズ |
|---|---|---|
| 非エンジニアのアプリ作成者 | 社内業務担当者、PM/PdM、小規模事業者、業務改善担当者 | 自然文でアイデアを入力し、AIの質問に答えながら、非技術者向けの表現で成果物を確認したい |
| 生成CRUDアプリの管理者 | 生成された業務アプリでデータを管理する担当者 | データの登録、一覧、詳細確認、編集、削除を分かりやすく行いたい |
| 生成CRUDアプリの一般利用者 | 生成された業務アプリに情報を入力・閲覧する社内利用者 | スマホやPCから迷わず入力・確認したい |
| プラットフォーム運用者 | AIプロバイダー、GitHub、AWS、セキュリティ、監査を管理する人 | 連携設定、権限、承認履歴、デプロイ履歴を安全に管理したい |
| 技術レビュアー | 必要に応じて設計・コード・テスト・デプロイ前チェックを確認する人 | 要件、設計、タスク、差分、テスト結果、リスクを追跡可能にしたい |

## 成功指標

| 指標 | 目標 |
|---|---|
| アイデアから要件整理までの到達 | 非エンジニアがプロジェクトを作成し、アイデア入力、質問回答、要件確認まで進められる |
| 承認の安全性 | 要件承認やデプロイ承認なしに、重要な工程が進まない |
| MVP範囲の制御 | 初期MVPでは生成対象を社内向けCRUDアプリに限定できる |
| デプロイ準備の分かりやすさ | データベース設定や外部公開条件などの不足を非技術者向けに説明できる |
| 監査性 | 承認、GitHub操作、デプロイ判断、デプロイ結果が記録される |
| 理解しやすさ | 技術用語やエラーが、ユーザーに分かる表現で要約される |

## 将来構想

AppRailは、最終的には業務アイデアを安全にアプリ化するための汎用的なAI開発プラットフォームを目指します。

長期的には以下を扱えるようにします。

- CRUDアプリ以外の複数アプリ種別。
- 要件整理とスコープ制御。
- 画面、データ、API、業務フローの設計生成。
- レビュー可能な差分付きコード生成。
- 自動テストとAIによる失敗要約。
- GitHubベースの変更管理。
- AWSデプロイとデプロイ前チェック。
- 運用ログ要約と改善提案。
- セキュリティ、コスト、外部公開、本番反映に対する人間の承認。

将来的な対象アプリ例:

- 社内向けCRUDアプリ。
- 業務ポータル。
- 予約・申請ワークフロー。
- 承認フロー付き業務アプリ。
- 分析・レポーティングダッシュボード。
- 顧客向けWebアプリ。
- 軽量な業務自動化ツール。

## MVPスコープ - 含めるもの

| 機能 | 内容 | 理由 |
|---|---|---|
| プロジェクト作成 | アプリ開発プロジェクトを作成・管理する | 成果物、承認、デプロイ履歴の単位を作るため |
| 自然文アイデア入力 | 作りたいアプリを自然文で入力する | 最初から技術要件を書かなくても始められるようにするため |
| AIによる質問生成 | 目的、利用者、スコープ、画面、データ、完成条件、リスクを質問する | 曖昧なまま設計・実装へ進まないようにするため |
| 要件整理 | 入力内容と回答から要件を整理する | ユーザーがAIの理解内容を確認できるようにするため |
| MVPスコープ定義 | 含める機能と含めない機能を分ける | スコープ拡大を防ぐため |
| 画面一覧生成 | 画面名、目的、主な操作、関連データを提示する | 業務フローを確認しやすくするため |
| データモデル生成 | PostgreSQL + Prisma前提のデータ構造を生成する | CRUDアプリ生成の土台にするため |
| 実装タスク分解 | 設計を実装可能なタスクへ分解する | AIが何を作るのか追跡できるようにするため |
| 承認ゲート | 要件、設計、実装差分、テスト結果、デプロイを確認・承認する | 人間が重要な判断を制御するため |
| GitHub連携 | リポジトリ、ブランチ、Pull Request、差分確認を扱う | 生成コードを追跡・レビューできるようにするため |
| Next.js CRUDテンプレート生成 | 選定スタックで動くWebアプリの土台を生成する | MVPの実装対象を明確にするため |
| テスト実行と要約 | テストを実行し、失敗理由を分かりやすく要約する | 品質状態を非技術者にも理解できるようにするため |
| デプロイ前チェック | 設定、ビルド、DB接続、外部公開条件を検査する | 危険または不完全なデプロイを防ぐため |
| AWS Amplifyデプロイ | 承認済みアプリをAWS Amplify Hostingへデプロイする | アイデアから公開URLまでの流れを完結させるため |
| 監査履歴 | 承認、GitHub操作、デプロイ履歴、AIログ要約を記録する | 後から判断と操作を追跡できるようにするため |

## MVPスコープ - 含めないもの

| 対象外 | 理由 | 将来対応 |
|---|---|---|
| モバイルネイティブアプリ生成 | MVPはレスポンシブWebアプリに集中するため | アプリ種別拡張時に検討 |
| デスクトップアプリ生成 | 初期のAI-DLCフロー検証には不要なため | アプリ種別拡張時に検討 |
| 大規模SNSやリアルタイムアプリ | 初期MVPとして複雑すぎるため | 専用テンプレートで検討 |
| 複雑な基幹業務システム | 権限、連携、運用要件が重いため | エンタープライズ向けに検討 |
| 金融・医療などの高リスク領域 | 専用のコンプライアンスと安全設計が必要なため | 規制領域対応として別途検討 |
| あらゆる技術スタックへの対応 | MVPでは安全性と検証容易性を優先するため | 将来のスタック拡張で検討 |
| DynamoDB生成 | MVPではPostgreSQL + Prismaに統一するため | データストア拡張時に検討 |
| 無承認の本番反映 | 人間の承認モデルに反するため | 対応予定なし |
| 無承認のAWSリソース削除 | 危険度が高いため | 対応予定なし |
| GitHubなしの本番運用デプロイ | MVPではGitHubによる追跡性を重視するため | 明確な理由がある場合のみ将来検討 |
| 完全な運用監視プラットフォーム | MVPではログ要約と改善提案の境界定義に絞るため | Operations機能として拡張 |
| 複雑なロールベース権限管理 | MVPでは権限モデルを簡潔に保つため | 将来の権限モデルで検討 |

## 主要ユーザージャーニー

### プロジェクト作成と要件整理

1. ユーザーがプロジェクトを作成する。
2. 作りたいアプリのアイデアを自然文で入力する。
3. AIが不足情報を質問する。
4. ユーザーが質問に回答する。
5. AIが要件とMVPスコープを整理する。
6. ユーザーが要件を確認し、承認する。

### 設計とタスク準備

1. AIが画面設計とデータ設計を生成する。
2. ユーザーが設計概要を確認する。
3. ユーザーが承認または修正依頼を行う。
4. AIが実装タスクへ分解する。

### 実装、レビュー、テスト

1. AIがNext.js CRUDテンプレートまたはタスク単位の変更を生成する。
2. GitHub連携がブランチとPull Requestを作成する。
3. AppRailが差分、リスク、スコープ逸脱を要約する。
4. ユーザーが実装差分を承認する。
5. テストを実行し、失敗内容を要約する。
6. ユーザーがテスト結果を確認する。

### デプロイと改善

1. デプロイ前チェックで必要設定と安全条件を確認する。
2. ユーザーがデプロイを承認する。
3. AWS Amplifyへデプロイする。
4. ユーザーがデプロイ結果とURLを確認する。
5. AppRailが履歴を記録し、改善候補を提示できるようにする。

## リスクと未決定事項

| リスク / 未決定事項 | 影響 | 現在の方針 |
|---|---|---|
| 非エンジニアが成果物を十分理解しないまま承認する可能性 | 誤ったアプリや危険な設定が進む可能性 | 非技術者向け要約、明示的な承認、詳細表示を併用する |
| AIが承認済み範囲を超えて実装する可能性 | 不要な機能やリスクが増える | スコープ逸脱を検知し、承認前に表示する |
| 認証なしの外部公開 | データ漏えいにつながる | 認証またはアクセス制限なしの外部公開をブロックする |
| AWSコストやリソース作成への不安 | 運用・費用リスクがある | 課金やリスクのある操作は明示的な確認を必要にする |
| 認証プロバイダーが未決定 | 実装計画に影響する | Auth.jsやCognitoなどの具体選定前に、provider-neutralな認証契約を維持する |
| MVP範囲が広くなりすぎる可能性 | 初期リリースが遅れる | 社内向けCRUDアプリに絞り、まず一連の流れを成立させる |
