# ユーザーストーリー 日本語確認版

## このファイルの位置づけ

このファイルは、`stories.md` の説明確認用日本語版です。ストーリーID、Epic構成、MVP/Post-MVPの範囲は原本と揃えています。正式な原本は `stories.md` とし、このファイルはレビューしやすくするための補助資料です。

## ストーリー生成方針

- **主役ペルソナ**: 社内向けCRUDアプリを作りたい非エンジニア
- **生成アプリ側ユーザー**: 管理者と一般利用者の基本操作を含める
- **分解方法**: ユーザージャーニーと機能単位のハイブリッド
- **受け入れ基準**: Given / When / Then とチェックリストの併用
- **粒度**: 実装とテストに直接つなげやすい小さめのストーリー
- **MVPのライフサイクル範囲**: 企画、要件整理、設計、コード生成、テスト、デプロイ、ログ確認、改善提案
- **承認ゲート**: 要件、設計、実装差分、テスト結果、デプロイを個別ストーリー化
- **セキュリティとPBT**: 横断ストーリーと関連ストーリーの受け入れ基準に併用
- **Post-MVP**: MVPとは別枠で短く記録

## MVPストーリー

### Epic 1: プロジェクト開始とアイデア入力

#### US-001: 開発プロジェクトを作成する

非エンジニアのアプリ作成者として、新しい開発プロジェクトを作成したい。そうすることで、アプリのアイデア、生成成果物、承認、デプロイ履歴を1つの場所で管理できる。

**受け入れ基準**

Given: ユーザーがプロジェクト作成画面にいる。  
When: プロジェクト名を入力して作成する。  
Then: システムはプロジェクト用の作業領域と初期状態を作成する。

チェックリスト:

- [ ] プロジェクト名とアプリ概要を入力できる。
- [ ] プロジェクト状態が作成される。
- [ ] 生成成果物がプロジェクト単位で管理される。
- [ ] アプリ種別は社内向けCRUDアプリに制限される。
- [ ] UIは詳細表示を除き、技術的なセットアップ用語を前面に出さない。

#### US-002: 自然文でアプリアイデアを入力する

非エンジニアのアプリ作成者として、作りたいアプリを自然文で説明したい。そうすることで、最初から技術的な要件を書く必要がなくなる。

**受け入れ基準**

Given: ユーザーがプロジェクトを作成済みである。  
When: 予約管理アプリや在庫管理アプリなどのアイデアを入力する。  
Then: システムはそのアイデアをAI-DLC分析の入力として保存する。

チェックリスト:

- [ ] 業務で使う自然な言葉で入力できる。
- [ ] 保存されたアイデアが後続の要件と追跡できる。
- [ ] UIは次のステップがアイデアの具体化であることを示す。
- [ ] 空欄や曖昧な入力には、追加情報を促す分かりやすい案内を表示する。

#### US-003: AIが生成した要件確認質問に回答する

非エンジニアのアプリ作成者として、不足している要件をAIに質問してほしい。そうすることで、曖昧なアイデアのままアプリが作られることを防げる。

**受け入れ基準**

Given: ユーザーがアプリアイデアを送信済みである。  
When: AIが不足情報を検出する。  
Then: システムは目的、利用者、MVP範囲、画面、データ、完成条件、リスクに関する質問を提示する。

チェックリスト:

- [ ] 質問はGUIで回答しやすい形式で表示される。
- [ ] 回答は保存され、要件と紐づく。
- [ ] 選択肢に合わない場合に自由記述相当の回答ができる。
- [ ] 必須質問が未回答のまま進もうとした場合は検出される。

#### US-004: 整理された要件を確認する

非エンジニアのアプリ作成者として、整理された要件を確認したい。そうすることで、設計に進む前にAIの理解内容を検証できる。

**受け入れ基準**

Given: ユーザーが要件確認質問に回答済みである。  
When: AIが要件を整理する。  
Then: ユーザーはアプリの目的、利用者、含める機能、除外する機能、画面、データ、完成条件、リスクを確認できる。

チェックリスト:

- [ ] 要件は非技術者向けの言葉で表示される。
- [ ] 技術的な詳細は折りたたみ領域で確認できる。
- [ ] ユーザーは承認前に修正依頼できる。
- [ ] 要件は元のアイデアと回答内容を参照できる。

#### US-005: 要件を承認する

非エンジニアのアプリ作成者として、要件を明示的に承認したい。そうすることで、後続の設計と生成が合意済みの範囲に基づいて進む。

**受け入れ基準**

Given: 要件が表示されている。  
When: ユーザーが要件を承認する。  
Then: システムは承認を記録し、次のAI-DLCステージを有効にする。

チェックリスト:

- [ ] 承認には明示的なユーザー操作が必要である。
- [ ] 承認日時と承認者が記録される。
- [ ] 要件承認前のコード生成はできない。
- [ ] ユーザーは承認ではなく修正依頼を選べる。

### Epic 2: スコープ、設計、タスク準備

#### US-006: MVPスコープを確認する

非エンジニアのアプリ作成者として、MVPスコープを確認したい。そうすることで、最初に何を作り、何を作らないのか理解できる。

**受け入れ基準**

Given: 要件が承認済みである。
When: システムがMVPスコープを提示する。
Then: 含める機能と除外する機能が明確に分かれて表示される。

チェックリスト:

- [ ] MVPは社内向けCRUDアプリに制限される。
- [ ] 高リスクまたは複雑な機能はスコープ外として示される。
- [ ] Post-MVP項目がMVPの約束範囲に混ざらない。
- [ ] ユーザーは設計承認前にスコープ変更を依頼できる。

#### US-007: 生成された画面一覧を確認する

非エンジニアのアプリ作成者として、生成された画面を確認したい。そうすることで、意図した業務フローをアプリが支えられるか確認できる。

**受け入れ基準**

Given: MVPスコープが定義されている。  
When: AIが画面一覧を生成する。  
Then: 各画面には目的、主な操作、関連データが含まれる。

チェックリスト:

- [ ] 生成CRUDアプリの管理者と一般利用者の基本操作を画面一覧がカバーする。
- [ ] 画面説明は非技術者向けの言葉で書かれている。
- [ ] 各画面はユーザーのニーズまたは要件に紐づく。
- [ ] 明らかに必要なCRUD画面が不足している場合はレビュー対象として示される。

#### US-008: PostgreSQLとPrismaのデータモデルを確認する

非エンジニアのアプリ作成者として、簡略化されたデータモデルを確認したい。そうすることで、アプリが正しい業務情報を保存するか確認できる。

**受け入れ基準**

Given: 画面一覧と要件が利用可能である。  
When: AIがデータモデルを生成する。  
Then: PostgreSQLとPrisma前提のエンティティ、フィールド、リレーション、制約が表示される。

チェックリスト:

- [ ] MVPのデータモデルはPostgreSQL + Prismaに限定される。
- [ ] データベース用語より先に業務上の名称が表示される。
- [ ] 必須項目と関連が確認できる。
- [ ] DynamoDBはMVPのデータモデルに含めない。

#### US-009: 設計を承認する

非エンジニアのアプリ作成者として、提案された設計を承認したい。そうすることで、未確認の構造から実装が始まることを防げる。

**受け入れ基準**

Given: 画面設計とデータ設計が表示されている。  
When: ユーザーが設計を承認する。  
Then: システムは承認を記録し、タスク分解またはコード計画へ進める。

チェックリスト:

- [ ] 設計承認は独立したストーリーおよびUI操作である。
- [ ] ユーザーは承認前に設計変更を依頼できる。
- [ ] 承認履歴には日時と承認者が含まれる。
- [ ] 承認前にセキュリティ警告を確認できる。

#### US-010: 実装タスク分解を確認する

非エンジニアのアプリ作成者として、実装作業を理解しやすいタスクに分けて確認したい。そうすることで、AIが何を作るのか追跡できる。

**受け入れ基準**

Given: 設計が承認済みである。  
When: AIが作業を分解する。  
Then: 各タスクには目的、対象領域、期待結果が含まれる。

チェックリスト:

- [ ] タスクは実装とテストに接続しやすい粒度である。
- [ ] タスクは技術的なファイル名より先にGUI向けの表現を使う。
- [ ] タスクは要件または画面と追跡できる。
- [ ] リスクの高いタスクは明示的なレビュー対象として示される。

### Epic 3: GitHub連携とコード生成

#### US-011: GitHubリポジトリを接続または作成する

プラットフォーム運用者または非エンジニアのアプリ作成者として、GitHubリポジトリを接続または作成したい。そうすることで、生成コードを追跡しレビューできる。

**受け入れ基準**

Given: プロジェクト設定が準備できている。  
When: ユーザーがGitHubを接続する。  
Then: システムはリポジトリを作成または接続し、生成変更用のブランチを準備できる。

チェックリスト:

- [ ] GitHub設定は最小権限で行われる。
- [ ] UIではPull Requestを「変更内容の確認」と説明する。
- [ ] リポジトリ操作とブランチ操作は履歴に記録される。
- [ ] GitHub設定エラーは非技術者向けに要約される。

#### US-012: Next.js CRUDテンプレートを生成する

非エンジニアのアプリ作成者として、Next.js CRUDテンプレートを生成したい。そうすることで、動作する土台から始められる。

**受け入れ基準**

Given: 要件、設計、GitHub設定が準備できている。  
When: テンプレート生成が実行される。  
Then: Next.js、TypeScript、Tailwind CSS、shadcn/ui、Prisma、PostgreSQL対応のプロジェクトが作成される。

チェックリスト:

- [ ] テンプレートは管理者と一般利用者の基本操作を支える。
- [ ] BackendはNext.js Route HandlersまたはHono互換構成を使う。
- [ ] ロックファイルと依存関係バージョンが追跡される。
- [ ] シークレットはハードコードされない。

#### US-013: 生成コードを確認する

技術レビュアーまたは非エンジニアのアプリ作成者として、生成コードを要約ビューで確認したい。そうすることで、承認前に何が変わったか理解できる。

**受け入れ基準**

Given: AIがコードを生成済みである。  
When: ユーザーがレビュー画面を開く。  
Then: 人間向けの要約、変更箇所、リスクメモ、必要に応じた技術詳細が表示される。

チェックリスト:

- [ ] 生の差分より先に変更要約が表示される。
- [ ] 技術詳細は確認できるが、初期表示の中心にはしない。
- [ ] スコープ拡張は強調表示される。
- [ ] セキュリティとテストへの影響が表示される。

#### US-014: 実装差分を承認する

非エンジニアのアプリ作成者として、実装差分を承認したい。そうすることで、承認済みの変更だけがGitHubへ反映される。

**受け入れ基準**

Given: 生成コードの変更が準備できている。  
When: ユーザーが実装差分を承認する。  
Then: システムは承認を記録し、変更をpushまたはPull Requestに含められるようにする。

チェックリスト:

- [ ] 承認は明示的である。
- [ ] 未承認の変更はpushされない。
- [ ] 承認履歴には承認者、日時、要約が記録される。
- [ ] ユーザーは却下または修正依頼を選べる。

#### US-015: AIによる未承認のスコープ拡張をブロックする

技術レビュアーとして、AIがスコープを拡張した場合に検出したい。そうすることで、意図しない機能が黙って追加されることを防げる。

**受け入れ基準**

Given: 生成コードに承認済み要件へ紐づかない動作が含まれる。  
When: システムが差分をレビューする。  
Then: その変更はスコープ拡張として表示され、承認または削除が必要になる。

チェックリスト:

- [ ] スコープ拡張は期待された変更と分けて表示される。
- [ ] 未承認で本番影響のある変更はブロックされる。
- [ ] ユーザーは受け入れ、却下、説明依頼を選べる。
- [ ] 監査ログに判断が記録される。

### Epic 4: テストと品質レビュー

#### US-016: テストを実行する

非エンジニアのアプリ作成者として、GUIからテストを実行したい。そうすることで、デプロイ前に生成アプリが動くことを確認できる。

**受け入れ基準**

Given: 実装変更が準備できている。  
When: ユーザーがテストを実行する。  
Then: システムは設定済みテストを実行し、成功、失敗、未実行の状態を表示する。

チェックリスト:

- [ ] 単体、統合、E2Eテストのカテゴリを表現できる。
- [ ] テスト状態は生ログを読む前に理解できる。
- [ ] 失敗時はAIによる簡潔な原因要約が表示される。
- [ ] ログ詳細を確認できる。

#### US-017: テスト結果を承認する

非エンジニアのアプリ作成者として、テスト結果を承認したい。そうすることで、テスト結果を理解したうえでデプロイへ進められる。

**受け入れ基準**

Given: テスト結果が利用可能である。  
When: ユーザーがテスト結果を承認する。  
Then: システムは承認を記録し、デプロイ準備を有効にする。

チェックリスト:

- [ ] テスト結果承認は独立したゲートである。
- [ ] 失敗テストは明示的な理由付きの上書きなしに承認できない。
- [ ] システムはリスクを非技術者向けに説明する。
- [ ] 承認履歴が記録される。

#### US-018: PBT対象ロジックを検証する

技術レビュアーとして、PBT対象のロジックを特定しテストしたい。そうすることで、バリデーション、データ変換、シリアライズ、正規化、スキーマ検証を堅牢にできる。

**受け入れ基準**

Given: 生成コードにPBT対象の関数が含まれる。  
When: テスト計画またはテスト実行が行われる。  
Then: システムは設定された範囲に対してproperty-based testを含める。

チェックリスト:

- [ ] PBTは純粋関数、バリデーション、データ変換、シリアライズ/デシリアライズ、正規化、AI生成成果物のスキーマ検証に適用される。
- [ ] MVPではUIコンポーネント、画面遷移、AWSデプロイ処理、GitHub API呼び出し、LLM呼び出し、E2EテストをPBTでブロックしない。
- [ ] PBT失敗時は、可能な範囲で再現可能な要約が表示される。
- [ ] テスト結果でPBTカバレッジを確認できる。

### Epic 5: デプロイ準備とAmplifyデプロイ

#### US-019: アプリ設定を行う

非エンジニアのアプリ作成者として、必要なアプリ設定を入力したい。そうすることで、シークレットを露出せずに生成アプリをサービスへ接続できる。

**受け入れ基準**

Given: デプロイ準備が開始される。  
When: データベース接続などの必須設定が不足している。  
Then: システムは不足内容と入力方法を説明する。

チェックリスト:

- [ ] 環境変数は「アプリ設定」として表示される。
- [ ] `DATABASE_URL` が未設定の場合、非技術者向けに説明される。
- [ ] 入力後のシークレットは全文表示されない。
- [ ] デプロイに影響する設定変更には確認が必要である。

#### US-020: デプロイ前チェックを実行する

非エンジニアのアプリ作成者として、公開前にデプロイ前チェックを実行したい。そうすることで、安全でない状態や未完了の設定をブロックできる。

**受け入れ基準**

Given: アプリがデプロイ可能な状態である。  
When: デプロイ前チェックが実行される。  
Then: システムはビルド状態、アプリ設定、DB接続、外部公開条件、セキュリティブロッカーを確認する。

チェックリスト:

- [ ] 認証またはアクセス制限なしの外部公開はブロックされる。
- [ ] シークレットのハードコードはブロッキング事項である。
- [ ] 過剰なIAM権限、危険なAWS削除、不適切なユーザーデータ送信、未承認の本番デプロイはブロッキング事項である。
- [ ] 軽微なセキュリティ指摘は警告として表示される。

#### US-021: デプロイを承認する

非エンジニアのアプリ作成者として、デプロイを明示的に承認したい。そうすることで、アプリが意図せず公開されることを防げる。

**受け入れ基準**

Given: デプロイ前チェックを通過している。  
When: ユーザーがデプロイを承認する。  
Then: システムはデプロイ承認を記録し、AWS Amplify Hostingへのデプロイを許可する。

チェックリスト:

- [ ] デプロイ承認は独立したゲートである。
- [ ] UIは外部公開が関係するかどうかを表示する。
- [ ] 課金影響の可能性がある操作には確認が必要である。
- [ ] 承認履歴には承認者、日時、対象環境が記録される。

#### US-022: AWS Amplify Hostingへデプロイする

非エンジニアのアプリ作成者として、GUIからAWS Amplify Hostingへデプロイしたい。そうすることで、生成アプリを公開できる。

**受け入れ基準**

Given: デプロイが承認済みで、GitHubが接続済みである。  
When: デプロイが開始される。  
Then: システムはAWS Amplify Hostingへのデプロイを開始し、状態を追跡する。

チェックリスト:

- [ ] AmplifyデプロイはGitHub連携を利用する。
- [ ] デプロイ状態がGUIに表示される。
- [ ] デプロイ失敗は非技術者向けに要約される。
- [ ] デプロイ履歴が記録される。

#### US-023: デプロイ結果と公開URLを確認する

非エンジニアのアプリ作成者として、デプロイ結果とURLを確認したい。そうすることで、アプリが利用可能か分かる。

**受け入れ基準**

Given: デプロイが完了している。  
When: ユーザーがデプロイ結果を表示する。  
Then: 成功または失敗、利用可能な場合は公開URL、次に推奨される操作が表示される。

チェックリスト:

- [ ] デプロイ成功時は公開URLが表示される。
- [ ] デプロイ失敗時は推定原因と次の操作が表示される。
- [ ] 外部アクセス制限の状態が見える。
- [ ] デプロイ結果は関連履歴に紐づく。

### Epic 6: デプロイ後レビューと改善サイクル

#### US-024: ログとエラーを分かりやすい言葉で確認する

非エンジニアのアプリ作成者として、デプロイや実行時のエラーを分かりやすい言葉で確認したい。そうすることで、何に対応すべきか理解できる。

**受け入れ基準**

Given: アプリにデプロイログまたは実行時ログがある。  
When: ユーザーが運用画面を開く。  
Then: システムは分かりやすい要約と詳細へのリンクを表示する。

チェックリスト:

- [ ] CloudWatch系のログは「エラー内容」などのユーザー向け表現で表示される。
- [ ] 要約に機密情報は表示されない。
- [ ] 要約はユーザーが行うべき操作と技術詳細を分けて表示する。
- [ ] 関連するデプロイ履歴またはテスト履歴にリンクされる。

#### US-025: 改善提案を作成する

非エンジニアのアプリ作成者として、デプロイ後にAIから改善提案を受け取りたい。そうすることで、次のAI-DLCサイクルを始められる。

**受け入れ基準**

Given: デプロイ結果、ログ、またはユーザーフィードバックが存在する。  
When: 改善分析が実行される。  
Then: システムは次サイクルへ入る前にレビュー可能な改善タスクを提案する。

チェックリスト:

- [ ] 提案は自動承認された作業とは分けて表示される。
- [ ] 各提案には理由と期待される効果が含まれる。
- [ ] 提案は新しいプロジェクトタスクまたは次サイクルの要件にできる。
- [ ] 高リスクな提案には明示的なレビューが必要である。

### Epic 7: 横断的な安全性、表現、監査、AIプロバイダー制御

#### US-026: GUI全体で非エンジニア向け表現を使う

非エンジニアのアプリ作成者として、技術的な概念をなじみのある言葉で表示してほしい。そうすることで、開発経験がなくても判断できる。

**受け入れ基準**

Given: 画面に技術的な概念が含まれる。  
When: UIがそれを表示する。  
Then: 最初に非エンジニア向けの表現を使い、必要な場合だけ技術詳細を表示する。

チェックリスト:

- [ ] Pull Requestは「変更内容の確認」と表示される。
- [ ] CI/CDは「自動チェック・自動公開」と表示される。
- [ ] Environment variablesは「アプリ設定」と表示される。
- [ ] CloudWatch logsは「エラー内容」と表示される。
- [ ] 技術詳細は折りたたみ領域で確認できる。

#### US-027: 承認と操作履歴を記録する

プラットフォーム運用者として、承認、GitHub操作、デプロイ履歴を記録したい。そうすることで、意思決定と変更を監査できる。

**受け入れ基準**

Given: ユーザーが承認または追跡対象操作を行う。  
When: 操作が完了する。  
Then: システムは実行者、日時、操作種別、要約を記録する。

チェックリスト:

- [ ] 要件、設計、実装差分、テスト結果、デプロイの承認が記録される。
- [ ] GitHub操作が記録される。
- [ ] デプロイ操作が記録される。
- [ ] AI内部ログは生ログ全文ではなく要約として保存される。

#### US-028: セキュリティブロッカーと警告を適用する

プラットフォーム運用者として、重大なセキュリティ問題はブロックし、軽微な指摘は警告にしたい。そうすることで、低リスクな指摘で過度に止めず、安全性を保てる。

**受け入れ基準**

Given: システムがセキュリティ指摘を検出する。  
When: 指摘が重大である。  
Then: システムは解消またはポリシーに沿った明示的対応が行われるまで進行をブロックする。

チェックリスト:

- [ ] シークレットのハードコードはブロッキング事項である。
- [ ] 過剰なIAM権限はブロッキング事項である。
- [ ] 認証またはアクセス制限なしの外部公開はブロッキング事項である。
- [ ] 危険なAWSリソース削除はブロッキング事項である。
- [ ] 不適切なユーザーデータ送信はブロッキング事項である。
- [ ] 未承認の本番デプロイはブロッキング事項である。
- [ ] 軽微な指摘は優先度つきの警告として表示される。

#### US-029: AI Provider Adapterを設定する

プラットフォーム運用者として、AIプロバイダー接続をアダプター経由で抽象化したい。そうすることで、MVPではAmazon Bedrockを優先しつつ、将来のプロバイダー拡張にも対応できる。

**受け入れ基準**

Given: プラットフォームがAI生成を必要としている。  
When: プロバイダー設定が行われる。  
Then: システムはプロダクトのワークフローにプロバイダー固有ロジックを直接埋め込まず、AI Provider Adapterを利用する。

チェックリスト:

- [ ] 初期プロバイダーはAmazon Bedrockを優先する。
- [ ] プロバイダー固有の詳細はアダプターの背後に分離される。
- [ ] 別プロバイダーを追加してもAI-DLCワークフロー画面を書き直す必要がない。
- [ ] BYO AI AccountはMVP必須ではないが、将来設計を妨げない。

## Post-MVP候補ストーリー

以下はMVPと同じ粒度では書かず、候補として短く記録する。

- PM-001: ユーザーとして、自分のAIアカウント/APIキーを接続したい。そうすることで、Bedrock、OpenAI、Anthropic、Geminiなどから選べる。
- PM-002: ユーザーとして、モバイルネイティブアプリ生成を使いたい。そうすることで、レスポンシブWeb以外のアプリも作れる。
- PM-003: プラットフォーム運用者として、App Runnerデプロイに対応したい。そうすることで、バックエンド中心のアプリをAmplify Hosting以外にも展開できる。
- PM-004: 管理者として、高度なロールベース権限管理を使いたい。そうすることで、より大きなチームで承認、デプロイ、設定変更の権限を管理できる。
- PM-005: ユーザーとして、より充実した運用ダッシュボードを使いたい。そうすることで、ログ、メトリクス、インシデント、改善タスクを1か所で管理できる。
- PM-006: ユーザーとして、追加のデータベースやデータストアに対応したい。そうすることで、必要に応じてDynamoDBなどを選択できる。
