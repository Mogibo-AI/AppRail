# Frontend Components - U1 Project, Workflow, and Data Access Core

## Scope

U1 does not own the full UI shell. U13 owns screen composition and visual implementation. This document defines only the U1-facing frontend contracts required for project creation, workflow state display, dashboard section availability, and next action routing.

## Component Boundary Principles

- Components consume U1 APIs; they do not contain workflow transition business rules.
- Components display U1's `NextAction` instead of computing their own next step.
- Components use `DashboardSection<T>` to distinguish available and unavailable sections.
- Components never call Prisma or repositories directly.
- Components route technical errors through U12 once U12 is available.
- Components must avoid exposing raw audit logs, job logs, database errors, or provider errors.

## ProjectCreateForm

**Purpose**

Collect the minimum project creation inputs required by U1.

**Fields**

| Field | Source | Validation |
|---|---|---|
| Workspace | Route/context or selector | Required; actor must have workspace access |
| Project name | User input | Required, 1 to 80 chars after normalization |
| App idea | User input | Required, 1 to 4000 chars |
| Target users | User input | Required, 1 to 1000 chars |
| Publication scope | Select/segmented control | Required; MVP allows internal or restricted external intent |
| App type | Hidden/default or advanced selector | Defaults to `internal-crud-web-app`; unsupported types blocked |

**State**

```ts
type ProjectCreateFormState = {
  workspaceId: ID;
  name: string;
  initialIdeaText: string;
  targetUsers: string;
  publicationScope?: PublicationScope;
  appType: "internal-crud-web-app";
  submitting: boolean;
  validationErrors: Partial<Record<"workspaceId" | "name" | "initialIdeaText" | "targetUsers" | "publicationScope" | "appType", string>>;
};
```

**API Integration**

```ts
POST /api/workspaces/{workspaceId}/projects
```

Expected service call:

```ts
createProject(input, actor)
```

**Interactions**

1. User enters required fields.
2. Form validates required fields and length before submit.
3. Form calls project creation API.
4. On success, UI navigates to `CreateProjectResult.project` dashboard URL.
5. On validation failure, field-level messages are shown from `UserFacingError.fieldErrors`.
6. On unsupported app type, UI explains that MVP supports internal CRUD apps only.

## NextActionPanel

**Purpose**

Show the next action generated by U1's workflow rules, including routing data.

**Props**

```ts
type NextActionPanelProps = {
  nextAction: NextAction;
};
```

**Interactions**

- Use `nextAction.target.href` for routing.
- Use `nextAction.target.unit` to decide which unit-owned UI area is being entered.
- If `nextAction.disabled` is true, do not navigate; show `disabledReason`.
- If `actionType = blocked`, link to `target.href` only if the blocker summary is available.
- If `actionType = approve`, route to the relevant approval UI owned by U11/U13 and use `requiredGate`.
- If `actionType = answer_questions`, route to U2 flow.
- If `actionType = run_job`, route to the appropriate unit-owned job action.

U13 must not infer next stage, gate, or route from enum values when `NextAction` already supplies them.

## Dashboard Section Components

All dependency-backed dashboard sections use:

```ts
type DashboardSection<T> =
  | { status: "available"; data: T; updatedAt: ISODateTime }
  | {
      status: "unavailable";
      unavailableReason: DashboardUnavailableReason;
      message: string;
      retryable: boolean;
      updatedAt?: ISODateTime;
    };
```

**UI Rule**

Unavailable sections render a neutral unavailable state for that section only. They do not replace the whole dashboard with an error page.

## ProjectArchiveDialog

**Purpose**

Let authorized users archive a project without physical deletion.

**Props**

```ts
type ProjectArchiveDialogProps = {
  projectId: ID;
  projectStatus: ProjectStatus;
  canArchive: boolean;
};
```

**API Integration**

```ts
POST /api/projects/{projectId}/archive
```

Expected service call:

```ts
archiveProject(projectId, actor)
```

**Rules**

- Hidden or disabled for archived projects.
- Shows that archive preserves history and does not delete records.
- Requires explicit confirmation.
- If archive fails with `PROJECT_ARCHIVED_READONLY`, refresh dashboard and show the archived state.

## API Endpoints Consumed by U13

| Endpoint | U1 Service Method | Purpose |
|---|---|---|
| `POST /api/workspaces/{workspaceId}/projects` | `createProject` | Create project and initial workflow |
| `GET /api/workspaces/{workspaceId}/projects` | `listProjects` | Project list within a workspace |
| `GET /api/projects/{projectId}` | `getProject` | Project detail |
| `GET /api/projects/{projectId}/dashboard` | `getProjectDashboard` | Dashboard read model with section availability |
| `GET /api/projects/{projectId}/workflow` | `getWorkflowState` | Workflow state |
| `POST /api/projects/{projectId}/workflow/advance` | `advanceStage` | Request stage advancement |
| `POST /api/projects/{projectId}/archive` | `archiveProject` | Archive project |

## Form Validation Rules

| Field | UI Validation | Formal Error Code | Server Still Required |
|---|---|---|---|
| Workspace | Required | `WORKSPACE_ID_REQUIRED` | Yes |
| Project name | Required, max length, trim whitespace | `PROJECT_NAME_REQUIRED`, `PROJECT_NAME_INVALID` | Yes |
| App idea | Required, max length | `PROJECT_IDEA_REQUIRED` | Yes |
| Target users | Required, max length | `PROJECT_TARGET_USERS_REQUIRED` | Yes |
| Publication scope | Required selection | `PROJECT_PUBLICATION_SCOPE_REQUIRED` | Yes |
| App type | Fixed MVP value | `PROJECT_TYPE_UNSUPPORTED` | Yes |

## Error Presentation

| Error Code | UI Handling |
|---|---|
| `WORKSPACE_ID_REQUIRED` | Show workspace selection error |
| `WORKSPACE_SCOPE_DENIED` | Show generic workspace access denied message |
| `WORKSPACE_MEMBERSHIP_REQUIRED` | Show generic workspace access denied message |
| `PROJECT_NAME_REQUIRED` | Show field-level message under project name |
| `PROJECT_NAME_INVALID` | Show field-level project name format/length message |
| `PROJECT_NAME_ALREADY_EXISTS` | Show field-level duplicate name message |
| `PROJECT_SLUG_ALREADY_EXISTS` | Ask the user to adjust the project name |
| `PROJECT_IDEA_REQUIRED` | Show field-level app idea message |
| `PROJECT_TARGET_USERS_REQUIRED` | Show field-level target user message |
| `PROJECT_PUBLICATION_SCOPE_REQUIRED` | Show selection-level message |
| `PROJECT_PUBLICATION_SCOPE_UNSUPPORTED` | Show MVP limitation message |
| `PROJECT_TYPE_UNSUPPORTED` | Show MVP limitation message |
| `PROJECT_NOT_FOUND_OR_DENIED` | Show generic access denied or not found message |
| `PROJECT_ARCHIVED_READONLY` | Refresh dashboard and show archived read-only state |
| `WORKFLOW_APPROVAL_REQUIRED` | Route to approval summary using `NextAction.target.href` |
| `WORKFLOW_BLOCKED_BY_SECURITY` | Route to blocker summary using `NextAction.target.href` |
| `AUDIT_OUTBOX_WRITE_FAILED` | Show retryable mutation failure message |

## Accessibility and Language Notes

- Primary labels use non-engineer wording.
- Technical enum values are hidden from primary UI.
- Required fields have visible labels and programmatic validation messages.
- Archive confirmation explains that history remains available.
- Next action text comes from U1 metadata and can be localized without changing U1 business logic.
- Dashboard unavailable sections should use clear wording such as "この情報は一時的に取得できません。"

## Security and PBT Notes

- SECURITY-05: Client validation is convenience only; server validation remains required.
- SECURITY-08: UI must not rely on hidden controls for authorization.
- PBT is not applied to visual rendering, but the form normalization, slug generation, next action generation, and dashboard section mapping functions consumed by UI are PBT-applicable through U1 code generation.
